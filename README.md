# MathScy: Domain-Specialized Mixture-of-Experts for Mathematical Reasoning

MathScy is a Mixture-of-Experts (MoE) system built on [DeepSeek-Math-7B-Base](https://huggingface.co/deepseek-ai/deepseek-math-7b-base) for mathematical conjecture generation, theorem proving, and formal verification. It uses domain-specialized QLoRA adapters trained via [Branch-Train-Mix (BTX)](https://arxiv.org/abs/2403.07816) with a sigmoid router for expert selection.

**Paper**: [`paper/mathscy.pdf`](paper/mathscy.pdf) (31 pages)
**Trained Model**: [ai4he/mathscy-moe-7b](https://huggingface.co/ai4he/mathscy-moe-7b) on HuggingFace
**Proved Conjectures**: [`results/proved_conjectures.md`](results/proved_conjectures.md) (NLP) | [`results/proved_conjectures.lean`](results/proved_conjectures.lean) (Lean 4)

## Architecture Overview

MathScy implements a three-phase pipeline:

```
Phase 1: Branch-Train-Mix (BTX)
  ArXiv corpus (6.5M statements) → Domain stratification → 7 QLoRA experts + 1 shared expert
  → Sigmoid router training → Assembled MoE system

Phase 2: Self-Play Theorem Prover (STP)
  Conjecturer generates hypotheses → Prover attempts proofs → Judge evaluates quality
  → Mutual training loop (multi-round)

Phase 3: Formal Verification
  Top conjectures → LLM autoformalization → Lean 4 type-checking → 4-tier verification
```

| S | Shared (all domains) | 14,000 | Cross-domain knowledge (2K per domain) |

### Step 4: MoE Expert Training (GPU)

#### Training a Single Expert

```bash

    --mode single \
    --domain algebra
```

**Training Configuration**:
| Parameter | Value |
|-----------|-------|
| Base Model | `deepseek-ai/deepseek-math-7b-base` |
| Quantization | 4-bit NF4 (QLoRA) |
| LoRA Rank (r) | 64 |
| LoRA Alpha | 128 |
| LoRA Dropout | 0.05 |
| Target Modules | q_proj, k_proj, v_proj, o_proj, gate_proj, up_proj, down_proj |
| Trainable Params | ~150M per expert (2.1% of base) |

**Output**: `models/expert_{domain}/final/` — 8 LoRA adapter directories (~600MB each)

### Step 5: Router Training & MoE Assembly

**Goal**: Train a sigmoid gating router and assemble all experts into a unified MoE system.

**Script**: `scripts/assemble_moe.py`

```bash
# Train router and evaluate
python3 scripts/assemble_moe.py --mode full

# Train router only
python3 scripts/assemble_moe.py --mode train-router

# Evaluate existing router
python3 scripts/assemble_moe.py --mode evaluate
```

The router is a lightweight sigmoid network that:
- Takes the input embedding and outputs a score for each of the 7 domain experts
- Selects the **top-2 experts** per input (plus the shared expert, always active)
- Uses **auxiliary-loss-free load balancing** to prevent expert collapse
- Achieves near-uniform expert utilization (load balance ratio: 1.26)

**Router Performance**:
| Metric | Value |
|--------|-------|
| Top-1 Accuracy | 76.3% |
| Top-2 Accuracy | 88.9% |
| Load Balance Ratio | 1.26 |

**Output**:
- `models/moe_assembled/router.pt` — Router weights (117KB)
- `models/expert_registry.json` — Expert-domain mapping
- `results/router_evaluation.json` — Full evaluation metrics

### Step 6: Conjecture Generation & STP Loop

**Goal**: Generate mathematical conjectures using multiple strategies and evaluate them via a Self-Play Theorem Prover loop.

**Script**: `scripts/evaluate_conjectures.py` | **Notebook**: `notebooks/04_conjecture_generation.ipynb`

#### Multi-Strategy Generation

```bash
python3 scripts/evaluate_conjectures.py --mode generate --provider groq
```

Five generation strategies:
1. **Pattern Interpolation** — Identify patterns and extend them
2. **Composition** — Combine theorems from different areas
3. **Boundary Exploration** — Test limits of known results
4. **Theorem Generation** — Direct novel theorem formulation
5. **Cross-Domain Analogy** — Transfer insights between domains

#### Self-Play Theorem Prover (STP) Loop

```bash
python3 scripts/evaluate_conjectures.py --mode stp --provider mistral
```

The STP loop:
1. **Generate**: Conjecturer proposes 4 conjectures per domain per round
2. **Prove**: Prover attempts to prove/disprove each conjecture
3. **Judge**: Independent evaluator scores quality, novelty, difficulty
4. **Iterate**: Results feed back into the next round (2+ rounds)

#### Full Pipeline (Generate + STP + Evaluate)

```bash
python3 scripts/evaluate_conjectures.py --mode full --provider mistral
```

**Supported LLM Providers**: `groq` (Llama-3.3-70b), `mistral` (mistral-small-latest), `gemini` (gemini-2.0-flash)

**Output**:
- `results/generated_conjectures.jsonl` — Multi-strategy conjectures
- `results/stp_{domain}_checkpoint.json` — Per-domain STP results
- `results/ranked_conjectures.json` — All conjectures ranked by quality
- `results/conjecture_evaluation_report.json` — Full evaluation report

The 4-tier verification pipeline:

## Model Architecture Details

### Branch-Train-Mix (BTX)

MathScy adapts the BTX paradigm from [Sukhbaatar et al., 2024]:

1. **Branch**: Fork the base model into 7 domain copies + 1 shared copy
2. **Train**: Fine-tune each copy independently on its domain data using QLoRA
3. **Mix**: Combine experts with a learned router that selects top-2 per input

The shared expert is always activated alongside the top-2 domain experts, providing cross-domain knowledge. This yields an effective 3-expert activation per input while maintaining 8 total experts.

### QLoRA Configuration

Each expert is a [QLoRA](https://arxiv.org/abs/2305.14314) adapter applied to the 4-bit NF4 quantized base model:

```
Base Model: deepseek-ai/deepseek-math-7b-base (6.9B parameters)
├── Quantized to 4-bit NF4 (~3.5GB)
├── Expert 1: LoRA adapter (r=64, 150M params, ~600MB)
├── Expert 2: LoRA adapter (r=64, 150M params, ~600MB)
├── ...
├── Expert 7: LoRA adapter (r=64, 150M params, ~600MB)
├── Shared:   LoRA adapter (r=64, 150M params, ~600MB)
└── Router:   Sigmoid gating network (117KB)
```

All 7 attention and MLP projection layers are targeted: `q_proj`, `k_proj`, `v_proj`, `o_proj`, `gate_proj`, `up_proj`, `down_proj`.

### Sigmoid Router

Unlike softmax-based routers (which enforce competition between experts), MathScy uses a **sigmoid router** where each expert is scored independently:

- Each expert gets a score in [0, 1] via sigmoid activation
- Top-2 experts are selected by score
- The shared expert is always included (not gated)
- Load balancing uses an auxiliary-loss-free approach (no extra loss term)

### Using Pre-Trained Models from HuggingFace

```python
from transformers import AutoModelForCausalLM, AutoTokenizer
from peft import PeftModel
import torch

# Load base model (4-bit quantized)
base_model = AutoModelForCausalLM.from_pretrained(
    "deepseek-ai/deepseek-math-7b-base",
    torch_dtype=torch.bfloat16,
    load_in_4bit=True,
    device_map="auto"
)
tokenizer = AutoTokenizer.from_pretrained("deepseek-ai/deepseek-math-7b-base")

# Load a domain expert (e.g., algebra)
model = PeftModel.from_pretrained(base_model, "ai4he/mathscy-moe-7b/expert_algebra/final")

# Generate
prompt = "Prove that every finitely generated module over a PID is a direct sum of cyclic modules."
inputs = tokenizer(prompt, return_tensors="pt").to(model.device)
outputs = model.generate(**inputs, max_new_tokens=512, temperature=0.7)
print(tokenizer.decode(outputs[0], skip_special_tokens=True))
```
